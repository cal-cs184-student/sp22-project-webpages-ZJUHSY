Serial optimization

The naive solution of particle simulation would have a time complexity of O(n^2). We use particle binning that only considers particles in neighboring bins for force interaction, which will decrease the simulation complexity to O(n) in theory.

Data Structure

We use the following data structure for binning. Each bin has a vector that stores the id of the particles in this bin, and a vector that stores the id of the neighboring bins. In the init_simulation function, we calculate the number of bins needed, initialize an array of bins, and calculate the neigh_bin_ids for each bin. Therefore, when simulation, we can just iterate through neigh_bin_ids to get the neighboring bins.

    class bin {

       public:

       std::vector<int> parts_ids;

       std::vector<int> neigh_bin_ids;

    };

Optimizations

The simulation time for 1M particles using naive particle binning is around 1200 seconds. After implementing the naive particle binning, we did some optimizations to further speed up serial simulation.

The first optimization is to apply force symmetrically on a pair of particles, that is, besides applying force on the current particle, we also apply force on the neighboring particle, so that we can decrease the number of times that apply_force is called. Also, when initializing the bins, we don’t need to count all 8 neighboring bins as neighbors for each bin, we can just consider the neighboring bins whose bin id is larger than the current bin. Therefore, when iterating through all the bins and their neighbors, we won’t compute force on the same pair of particles twice. The simulation time for 1M particles after this optimization is around 1000 seconds.

The second optimization is to only rebin particles that move to another bin instead of rebinning every particle. We know that a large portion of the particles stay in the same bin after one step of simulation, so only rebin particles that cross borner will reduce the simulation time. The simulation time for 1M particles after this optimization is around 830 seconds, which is our final result.

Shared Memory Algorithm, OpenMP

The particle simulation was then parallelized using OpenMP, which was built using the same technique. This is accomplished by assigning the particles to an extra vector list to track their motions, with the results being gathered from all threads and summed synchronously at the end.

Design choices

Since we parallelize our simulation_one_step in a shared memory environment, we adopted OpenMp method to do parallelization.

  1. Compute force: #pragma omp for

Omp for is a worksharing, parallel loops, loop carried dependencies method. We use it to split loop iterations among threads in a team. But we have tested methods of enabling nested parallel in each level of loops, which makes performance approximately slower than serial. To avoid excessive synchronization is key to success.

  1.  Move particles : synchronization for bins read and write

In the process of Compute Force, there is no memory conflict in terms of reading and writing particles. (Each index in the loop enables accessing different particles for computing the force). Therefore, we do not need synchronizations.

However, in the process of  Moving Particles and Rebining, different threads can end up accessing the same address. To be more specific, each loop includes a computation process of removing the current moved particle from its old bin and assigning it to the new bin. The push back operation in each new bin’s part_ids (See our data structure) requires a memory reallocation, freeing the previous memory chunk which stores the bin. Other threads might access or reallocate the unassigned space without a proper synchronization. In shared memory, threads communicate implicitly by writing and reading shared Variables, for example, race conditions. We conducted 2 ways to solve that problem:

  • Using Master thread. We can simply make this process executed by a single thread. However, the time is very slow, 3.3s for 1000 particles.
  • Using Critical to do synchronization: Use # omp critical to do mutual exclusion and allow only one thread at a time to enter the critical region (removing old bin and assigning new bin). Improving the efficiency of moving 1000 particles from 3.3s to 1.1s, about 3x speed up.
  • Using locks to avoid overhead. The Critical region only allows one thread to execute the whole block of code at one time. However, this is wasteful. To avoid race conditions, different threads can execute the critical region as long as they are accessing different elements of the bins data structure. Therefore, we can use locks for synchronization.

First, we initialize an array of locks for every bin. Then we give lock specific bin id to make sure memory read and write in the above critical region are independent. We use that as our final design plan. After this process, the original 1M particles on 68 threads from 330s to 88.5s, which is 4x speed up.

Finally, we found we could initial the locks on init_simulazation process rather than simulation_one_step. (There is no need to initialize the lock again in each thread, which makes the time complexity from o(n^2) to o(n), assuming n is the number of bins). Then our original 1M particles on 68 threads from 88.5s to 25s, which is another 3-4x speed up.

Time Break-up

We also break up the overall process into the computation process and synchronization process.  

                                    plots of fraction of time vs Pthreads

The synchronization process consists of the rebining process which causes race conditions, removing the moved particles into new bins and ejecting them from old bins as mentioned above. We need to use locks for this part.

We can see from the figure that the fraction of time of the synchronization is about 10% ~ 30%. Also, there is a reduction of the percentage the synchronization occupies when thread number is larger. This makes sense because there will be less probability of memory conflicts when there are more threads. (Each thread has a smaller range of index).
